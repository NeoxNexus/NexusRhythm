## 2. 关键技术决策

### 决策一：不把项目方向调整为“多平台工具”
- **背景**：参考项目 `Trellis` 的多平台和 CLI 产品化能力明显更强，存在被带偏成“平台型工具”的风险。
- **选择**：保留 NexusRhythm 的核心定位为“AI 项目节奏操作系统”，Claude Code 优先，不以多平台优先为近期目标。
- **原因**：项目当前最有辨识度的是 `ROADMAP + Phase + Mode + Debt + Distill` 这套节奏治理体系，而不是工具接入广度。
- **影响**：v1.0 前不把“支持多少平台”作为主成功标准。

### 决策二：未来演进采用“方法论内核 + 工程执行层”双层路线
- **背景**：当前很多能力已经写进规则和命令文档，但实际仍主要依赖 Agent 自觉执行。
- **选择**：不推翻现有方法论，优先补执行层，把关键命令脚本化，再逐步补任务层、热记忆层和并行层。
- **原因**：现有项目哲学已经足够强，真正的缺口是可执行性、可重复性和跨会话连续性。
- **影响**：未来的版本重点会先落在 `scripts/`、任务结构、记忆结构，而不是新增更多口号式命令。

### 决策三：引入 Task，但不替代 Phase
- **背景**：与 Trellis 对比后发现，NexusRhythm 只有阶段层，没有正式任务层，导致阶段内多个子任务、并发工作和 handoff 不够稳定。
- **选择**：未来引入 `.nexus/tasks/` 作为任务层，但保留 `Phase` 作为项目节奏主轴。
- **原因**：`Phase` 是 NexusRhythm 的产品差异化，不应被 `Task` 替代；正确方向是 Phase 管节奏，Task 管执行单元。
- **影响**：路线图里将 `v0.3` 定义为“任务层引入”，而不是整体架构改写。
