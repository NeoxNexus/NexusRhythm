## 2. 关键技术决策

### 决策一：先把 `/sync` 做成导航卡，而不是继续堆状态字段
- **背景**：Phase 0 已经确立了“规则给 AI 看，控制感给人看”的方向，但 `/sync` 仍主要在罗列状态字段。
- **选择**：保留轻量 banner，把 `/sync` 主体改成“当前步骤 / 原因 / 下一步”，技术状态后置。
- **原因**：这是把默认协作模式从理念推进到实际可感知行为的最小一步。

### 决策二：Phase 1 的 skill 试点选择 `doctor`
- **背景**：路线明确要求“只做一个代表性 workflow 的试点迁移”，但不能把范围拉成大重构。
- **选择**：迁移 `doctor`，而不是更复杂的 `review` 或 `phase-end`。
- **原因**：`doctor` 已有稳定脚本入口，最适合作为“命令 + 脚本 + skill 并存”的安全试点。

### 决策三：SessionStart 继续保持极简
- **背景**：新手体验优化容易把每次会话启动变成信息轰炸。
- **选择**：SessionStart 仍只显示 banner，完整导航卡只在显式 `/sync` 暴露。
- **原因**：这符合渐进式披露，也避免把 `CLAUDE.md` 和启动噪音一起做大。

## 7. 小复盘

**最大意外**：真正推动“零感知体验”的，不是新增一个大功能，而是把 `/sync` 这种已有入口改成更像导航卡。

**最大收获 / 教训**：如果要让规则主要给 AI 看，就必须持续把内部状态翻译成用户能立刻行动的人话，而不是默认把字段暴露给用户。

**下个阶段工作方式上想改进的地方**：继续用“小步可验证”的方式推进 Phase 2，不要同时启动 skill 迁移、统一入口和大文档重构。
