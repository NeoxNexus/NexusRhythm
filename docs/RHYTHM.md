# NexusRhythm — 开发规范 v2.1
> 📖 本文档是面向**人类**的完整规范说明。AI 行为约束详见根目录 `CLAUDE.md`。

---

## 0. 设计哲学

这套开发节奏有两个核心信念：

**第一：结构是自由的前提，不是阻碍自由的枷锁。**
有了明确的节奏和仪式，你才能放心地在该放飞的时候放飞（Vibe Sprint），在该严谨的时候严谨（三门禁），而不是整个过程都焦虑不安。

**第二：AI 是有独立判断的搭档，不是照单全收的执行者。**
当你作为人类提出一个存在技术债务或架构隐患的指令时，AI 必须主动质疑、提出证据、给出替代方案。你可以坚持，但AI不应沉默。

## I. 工作模式

**在每个阶段开始前，先选好模式。**

| 模式 | 名称 | 适用场景 | 纪律 | 典型时长 |
|:----:|------|----------|------|----------|
| **0** | Vibe 冲刺 | 周末实验、快速原型、算法探索 | 解除门禁，但产生债务必须标记 | 4–12 小时 |
| **1** | 标准模式 ⭐ | 核心功能、长期保留的代码 | SDD 先行 + TDD + 三门禁 | 1–7 天/阶段 |
| **2** | 生产模式 | 公开 API、安全敏感、有外部协作者 | 模式1 + 覆盖率≥80% + Reviewer | 1–4 周/阶段 |

> **绝大多数有价值的工作都在模式 1 进行。** 模式 0 是找回编程快感的安全阀，不是逃避纪律的借口。

## II. 核心原则（模式 1+ 不可妥协）

### 原则一：规范先行 + 测试先行（SDD + TDD 融合）

**先定义接口，再写实现。** 具体阶段状态机：

```
PLANNING → SPEC_READY → RED_TESTS → GREEN_CODE → GATE_CHECK → REVIEW → DONE
```

每个步骤都有明确的前置条件：
- `SPEC_READY`：SPEC 文档定义了接口、数据流、边界条件
- `RED_TESTS`：测试用例已写好且全部失败
- `GREEN_CODE`：所有测试通过，实现符合 SPEC
- `GATE_CHECK`：三门禁全部通过
- `REVIEW`：代码评审通过，CODE_REVIEW 文档已产出

### 原则二：ROADMAP.md 是唯一真相源

- 每次新会话必须首先读取 ROADMAP.md 的 YAML 头部
- 每个阶段结束必须更新 `Phase_Status` 和进度
- ROADMAP.md 的状态是当前开发状态的权威引用

### 原则三：三道强制门禁

提交代码前必须全部通过：
1. **类型检查 / 静态分析** — 零错误（mypy / eslint / rustc 等）
2. **构建成功** — 项目可以正常编译/运行
3. **全量测试 100% 通过** — 无跳过，无 skip

### 原则四：阶段结束仪式（5 步）

每个阶段完成后，必须执行（可使用 `/phase-end` 命令）：
1. 三门禁全部通过
2. 更新 ROADMAP.md 进度和状态
3. 产出 `docs/walkthroughs/WALKTHROUGH_PHASE_N.md`
4. 产出 `docs/reviews/CODE_REVIEW_PHASE_N.md`
5. 2 分钟小复盘 + 点子评审：复盘本阶段产生的新点子，审核通过后再同步到计划

### 原则五：严格向后兼容 + 最小性能影响

- 新功能不得对现有用户/集成/客户端产生破坏性变更
- 核心热路径的额外开销必须保持用户无感知水平
- 能走直接路径时，优先避免新增处理层

### 原则六：指令不是圣旨——主动思考与辩驳

当人类建议存在明显问题时，AI 必须：
1. **列出优缺点与证据**（数据 > 直觉）
2. **给出反例或替代方案**
3. **明确建议是否否决**

这不是对抗，这是负责任的协作。

### 原则七：点子先收集，后准入

- 执行过程中出现的好点子，优先记录到 `docs/IDEA_BACKLOG.md`
- 阶段结束前必须经过 `/idea-review`
- 只有 `Approved Now` 或 `Approved Later` 的点子才能进入 `ROADMAP.md`、阶段计划或 ADR

## III. 高杠杆习惯

### 深度参考研究优先
大型功能或库集成前：先完整理解标杆实现 → 提炼核心机制 → 设计融合方案，而非直接让 AI 从零生成。

### 架构解耦红线
- 外部协议/客户端适配：用 **Strategy / Adapter** 模式
- 特定场景逻辑：通过 **Plugin / 扩展点** 实现（绝不硬编码进核心）
- 数据流处理：优先采用**责任链**等成熟模式

### Vibe Sprint 规则（每完成 3 个阶段可解锁一次）
- 连续 4–12 小时纯写代码，不写文档，不跑测试
- 提交"毛坯版"后，48 小时内补齐文档和清理
- 超时未还债：`Pending_Debt: true`，所有 AI 会话拒绝新功能请求

## IV. 记忆管理系统

### 三层记忆架构

| 层 | 类型 | 命令 | 存储位置 | 更新频率 |
|:--:|------|------|----------|----------|
| L0 | 项目日志 Journal | `/journal` | `docs/journal/YYYY-MM-DD.md` | 每天 |
| L1 | 决策记录 ADR | `/decision [主题]` | `docs/decisions/ADR-NNN-xxx.md` | 每次重要决策 |
| L2 | 活知识 Rules | `/distill` | `.claude/rules/lessons.md` | 每 3 个阶段蒸馏 |

### 工作流诊断

- `/doctor`：检查脚手架安装、hooks、commands、阶段文档和点子治理闭环是否健康
- `/doctor` 默认只诊断，不隐式修复

### 反馈闭环
```
做事 → Journal(记录坑) → /distill(蒸馏) → .claude/rules/(自动加载)
                                                      ↓
                                            下次 AI 会话主动避坑
```

### 点子治理闭环
```
执行中冒出点子 → docs/IDEA_BACKLOG.md（原始收集）
               ↓
         /idea-review（客观评审）
               ↓
Approved Now / Approved Later → 计划文档 / ROADMAP / ADR
Reject / Hold                → 保留在 backlog，不进入阶段承诺
```

- **未审查点子禁止直接写进 ROADMAP.md**
- backlog 是收集区，不是承诺区
- 只有通过评审的点子才能影响阶段计划和优先级

## V. 阶段启动快速检查清单（模式 1）

```
☐ 读取并理解 ROADMAP.md 当前状态
☐ Pending_Debt: false（如果为 true，先清债）
☐ 上一阶段的 WALKTHROUGH 和 CODE_REVIEW 已完成
☐ 明确本阶段唯一核心目标
☐ 完成 SPEC 文档（接口、数据流、边界条件）
☐ 编写会失败的测试用例（红灯）
☐ 编码直至测试全绿
☐ 执行三门禁检查
☐ 执行阶段结束仪式
```
