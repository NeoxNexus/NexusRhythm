## 6. 发现的问题

### 🔴 必须修复（BLOCKING）

### 🟡 建议改进（NON-BLOCKING）
- 如果后续支持更多 CI 提供方，建议先抽象“自动化基线资产”的目录与检测规则，避免 `install.sh` 和 `doctor` 各自维护一份硬编码列表
- Phase 2 后续若继续扩 skills，应先定义“哪些 workflow 仍是 command-first，哪些正式迁到 skill-first”，否则容易出现入口边界漂移
- 当前 workflow 只覆盖 GitHub Actions；这是合理收敛，但后续对外产品化前仍需明确其他平台的支持策略

### 🟢 Nits（可选）
- 可以在后续迭代里给 CI workflow 增加更明确的 step 名称或失败提示，方便下游用户快速定位哪一道门禁失败
