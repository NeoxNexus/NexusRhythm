# NexusRhythm Product Roadmap v0.2 -> v1.0

**文档日期**：2026-03-08
**定位**：面向 Claude Code 优先、强调项目开发节奏治理的 AI 工程框架
**相关分析**：`docs/reviews/competitive-analysis.md`
**执行分解**：版本路线的执行拆解与实时状态请以 `ROADMAP.md` 为准。

---

## 1. 产品方向

### 1.1 愿景

把 NexusRhythm 从“高质量方法论模板”升级为“可执行、可升级、可复用的 AI 项目节奏操作系统”。

### 1.2 不变的核心

以下内容应被视为产品内核，不应在演进过程中丢失：

- `ROADMAP.md` 作为项目状态真相源
- `Phase` 阶段状态机
- `Mode 0 / 1 / 2` 工作模式
- `Pending_Debt` 技术债治理
- `Spec -> Red Tests -> Green Code -> Gate -> Review` 质量闭环
- `Journal -> ADR -> Walkthrough -> Review -> Distill` 记忆闭环
- Vibe Sprint 作为纪律与创造力之间的安全阀

### 1.3 未来版本的判断标准

一个版本是否成功，不看命令数量，而看三件事：

1. 关键流程是否更可执行
2. 跨会话协作是否更连续
3. AI 是否更难偏离项目节奏

---

## 2. 产品原则

### 原则一：方法论优先，自动化跟进

先保护核心节奏设计，再补工程化执行层，不为了工具感牺牲产品灵魂。

### 原则二：少数关键命令深做，不堆命令数量

优先做深：

- `/sync`
- `/phase-start`
- `/gate-check`
- `/phase-end`
- `/distill`

### 原则三：默认单线程，复杂任务才并行

并行能力应是高级模式，而不是默认模式。

### 原则四：上下文预算可解释

任何常驻规则、任务上下文、记忆文件，都应说明为什么要加载、何时加载、开销大概多少。

### 原则五：先支持 Claude Code，后考虑扩平台

v1.0 之前不把“多平台支持”设为主目标。

### 原则六：规则给 AI 看，控制感给人看

NexusRhythm 的核心规则主要是写给 AI 的，不是要求普通用户先学会 hooks、skills、subagents、状态机术语后才能开始。

但“AI 看得懂”不等于“人类完全看不懂”。产品应该默认向人展示：

- 当前步骤
- 为什么是这一步
- 下一步是什么

而不是让新手一开始就背完整命令表。

---

## 2.1 零感知北极星

从 v0.3 开始，产品演进应持续满足一个判断标准：

> 普通用户主要通过“表达意图”来驱动系统，AI 在底层负责把意图映射到 Discovery、Delivery、命令、文档和门禁。

这条北极星意味着：

- 对新手：降低术语面和命令面
- 对熟练用户：保留显式控制和深度能力
- 对系统本身：不能因为“易用”就放松节奏纪律

建议将该方向视为一条贯穿 v0.3 到 v1.0 的产品主线，而不是一次性的文档优化任务。

## 4. 分版本计划

### v0.2 — 命令脚本化基础

**目标**：把最关键的开发流程变成确定性动作，而不是仅靠 Agent 理解文案执行。

**核心交付物**：

- 新增 `scripts/` 目录
- 实现 `sync` 状态解析脚本
- 实现 `phase-start` 前置检查脚本
- 实现 `gate-check` 技术栈检测与执行脚本
- 实现 `phase-end` 基础编排脚本
- 实现 `distill` 原始素材扫描与去重脚本

**验收标准**：

- 五个核心命令都能在空白新项目中稳定运行
- 命令失败时有清晰错误信息
- 关键状态更新不依赖手工编辑 YAML

**不做**：

- 不做并行
- 不做多平台
- 不做复杂可视化

### v0.3 — 任务层引入

**目标**：在不破坏 `Phase` 模型的前提下，引入更细颗粒任务管理。

**建议结构**：

```text
.nexus/
├── tasks/
│   └── YYYY-MM-task-name/
│       ├── task.yaml
│       ├── prd.md
│       ├── handoff.md
│       ├── implement.jsonl
│       ├── review.jsonl
│       └── debug.jsonl
```

**核心交付物**：

- 任务创建命令
- 任务状态字段设计
- 阶段与任务的映射关系
- 任务上下文文件规范

**验收标准**：

- 一个阶段内可稳定承载多个任务
- 每个任务都有独立 handoff
- reviewer 能识别任务级上下文

### v0.4 — 热记忆层

**目标**：补齐当前系统在短期工作记忆上的缺口。

**建议结构**：

```text
.nexus/
├── memory/
│   ├── today.md
│   ├── active-tasks.json
│   ├── blockers.md
│   └── handoff.md
```

**核心交付物**：

- 自动 session-end flush
- 任务完成/中断时更新热记忆
- 新会话优先读取热记忆摘要

**验收标准**：

- 中断后可在 2 分钟内恢复上下文
- 今天的工作状态不必依赖翻 journal 回忆
- handoff 信息具备最小可执行性

### v0.5 — 评审路由升级

**目标**：让评审从单一 reviewer 进化为 maker-checker 体系。

**新增角色建议**：

- `implementer`
- `reviewer`
- `security-reviewer`
- `performance-reviewer`
- `debt-collector`

**核心交付物**：

- 评审触发矩阵
- 高风险改动的安全检查规则
- 热路径变更的性能检查规则

**验收标准**：

- 核心逻辑变更时自动触发更严格评审
- 评审报告具备分类输出
- `/distill` 可直接消费评审教训

### v0.6 — 可选并行模式

**目标**：为复杂阶段提供可选并行执行能力。

**核心交付物**：

- worktree 配置文件
- 并行任务启动命令
- worktree 初始化与清理脚本
- reviewer/checker 在隔离工作树运行

**启用条件**：

- 当前阶段存在 2 个以上独立子任务
- 用户明确要求并行
- 任务之间无强耦合写冲突

**验收标准**：

- 并行任务互不污染
- 任务完成后可独立回收与归档
- 主工作树状态保持干净

### v0.7 — CLI 初版

**目标**：让 NexusRhythm 从模板升级为可安装工具。

**建议命令**：

- `nexus init`
- `nexus update`
- `nexus doctor`
- `nexus sync`
- `nexus phase start`
- `nexus gate check`
- `nexus distill`

**核心交付物**：

- CLI 初版
- 版本号管理
- 安装与升级流程
- 自检与迁移提示

**验收标准**：

- 新项目 5 分钟内可完成初始化
- 升级时不会破坏用户手写内容
- `doctor` 能识别缺失文件、错误状态、坏配置

### v0.8 — Preset 与模板生态

**目标**：降低不同技术栈接入门槛。

**优先支持的 preset**：

- TypeScript Node
- Python FastAPI
- Next.js
- Go API

**核心交付物**：

- 技术栈 preset
- 默认 gate-check 命令映射
- 文档模板变体

**验收标准**：

- 主流项目无需大幅手改就能落地
- 新用户接入成本明显低于手配模板

### v0.9 — 稳定性与可观测性

**目标**：让系统更可靠、更可解释。

**核心交付物**：

- `docs/context-budget.md`
- 命令执行日志
- distill 结果预览
- 配置兼容性检查
- 版本迁移文档

**验收标准**：

- 用户能知道上下文预算花在哪
- 命令失败可追踪
- 升级风险可预估

### v1.0 — 节奏操作系统完成态

**目标**：形成稳定、可复用、具备明显差异化的产品闭环。

**v1.0 应具备的能力**：

- 项目状态机完整可执行
- 任务层与阶段层协同稳定
- 记忆蒸馏与热记忆协同稳定
- 质量门禁、评审、清债闭环稳定
- CLI、preset、doctor、迁移机制可用

**v1.0 的产品主张**：

不是“另一个 AI coding 模板”，而是“一个让 AI 遵守项目开发节奏的操作系统”。

## 6. 风险与应对

### 风险一：系统过重，失去“Clone 即用”

**应对**：

- 所有高级能力都做成可选层
- 保持最小基础模式可直接使用

### 风险二：过度追 Trellis，丢失自身定位

**应对**：

- 每个新能力都回答一个问题：是否强化了“开发节奏治理”
- 如果只是让工具看起来更像平台，但不强化节奏，就降低优先级

### 风险三：脚本化后灵活性下降

**应对**：

- 命令提供 `--dry-run`
- 保留手动 override，但默认走脚本

### 风险四：并行能力引入心智负担

**应对**：

- 并行只在高级模式启用
- 单任务默认仍使用简单流

## 7. 建议的近期执行顺序

### 第一批

- 实现 `sync`、`phase-start`、`gate-check`
- 定义 `.nexus/tasks/` 结构
- 增加最小 `today.md` 与 `active-tasks.json`

### 第二批

- 实现 `phase-end`、`distill`
- 增加 reviewer/security/performance 路由
- 设计 `doctor` 命令

### 第三批

- worktree 并行原型
- CLI 初版
- 技术栈 preset

---

## 8. 一句话版本路线

v0.2 到 v0.4 解决“能不能稳定执行”，v0.5 到 v0.7 解决“能不能形成产品”，v0.8 到 v1.0 解决“能不能规模化复用”。
